## Resumen Ejecutivo

Se diagnosticaron y corrigieron 3 bugs críticos que impedían que el dashboard de Analytics funcionara con los 20,413 clientes reales importados. También se creó data mock de reservas para probar el funcionamiento completo del sistema.

## Problemas Encontrados y Fixes

### Bug 1: Columnas inexistentes en la API de Analytics
**Problema:** La API consultaba 5 columnas que NO existen en `customer_stats`:
- `cancellations` ❌
- `total_party_size` ❌
- `avg_party_size` ❌
- `marketing_opt_in` ❌
- `blacklisted` ❌

**Causa:** El código se escribió asumiendo un schema ideal que no coincide con la tabla real en Supabase.

**Columnas reales de `customer_stats`:**
```
id, customer_id, total_visits, total_spent, last_visit_date,
no_show_count, is_recurring, loyalty_tier, updated_at
```

**Fix:** Reescribí la query del API para usar solo columnas existentes. Reemplacé métricas que no existen por equivalentes reales:
- `avgPartySize` → `avgSpendPerVisit` (total_spent / total_visits)
- `marketingOptIn` → eliminado (no hay columna)
- `blacklisted` → eliminado
- `totalCancellations` → eliminado

**Commits:** `d87f744`, `7a74a62`, `9b9cfbd`, `040c704`

### Bug 2: Límite de 1000 filas en Supabase
**Problema:** El dashboard solo mostraba ~1000 registros de los 20,413 importados.

**Causa:** Supabase REST API (PostgREST) trunca automáticamente a 1000 filas por request, incluso con `Range` headers que piden más. La función `fetchAll()` usaba batches de 2000 y una condición de parada incorrecta (`data.length === batchSize`), que nunca se cumplía porque siempre recibía ≤1000.

**Verificación directa:**
```
GET customer_stats con Range: 0-1999 → Content-Range: 0-999/* (solo 1000)
GET customer_stats con Range: 2000-3999 → Content-Range: 2000-2999/* (solo 1000)
```

**Fix:** 
- `fetchAll()` ahora usa batches de 999 registros
- Condición de parada: `if (data.length <= batchSize) break` en vez de `data.length === batchSize`
- Offset incrementa en `batchSize + 1`
- Para contactabilidad (celular/email), reemplacé la descarga de 20K customers por queries `count=exact` que son eficientes

### Bug 3: Segmentación con nombres incorrectos
**Problema:** `SegmentBreakdown.tsx` tenía `TIER_CONFIG` con nombres que no existen en el enum de la DB (`habitual`, `frecuente`, `prospecto`).

**Enum real en Supabase:**
```sql
loyalty_tier ENUM: 'none', 'new', 'occasional', 'regular', 'vip'
```

**Distribución real de los 20,413 clientes:**
| Tier | Cantidad |
|------|----------|
| none | 14,661 |
| new | 5,648 |
| occasional | 102 |
| regular | 2 |
| vip | 0 |

**Fix:** `TIER_CONFIG` alineado con el enum real. Métrica de "retenibles" ahora es `occasional + new`.

```

├── Analytics View
│   ├── KPIStatsBar (6 cards: Total, Recurrentes, Activos30d, Activos90d, Visitas, Gasto/Visita)
```

## Lecciones Aprendidas

1. **Supabase REST API max 1000 rows** — PostgREST trunca silenciosamente. Siempre paginar en batches de 999 con offset incremental.
2. **Verificar columnas reales antes de codificar** — El API asumía columnas que nunca existieron en `customer_stats`.
3. **URL de Vercel puede cambiar** — `web-pdqveiuty` → `web-rosy-one-64`. Siempre verificar con API.
4. **DDL en Supabase solo manual** — `ALTER TYPE` no se puede ejecutar por REST API. Alejandro lo hace desde el Dashboard.
5. **psycopg2 no funciona en WSL** — IPv6 incompatibility con Supabase. Workaround: REST API con `requests`.
